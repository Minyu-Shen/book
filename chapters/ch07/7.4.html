7.4 算法层-逻辑层-物理层的仿真架构

7.2节和7.3节分别介绍了实体建模和仿真循环。对于库存管理这类简单场景，一个类加一个循环就能实现完整的仿真器。然而，当我们面对车队调度、需求响应公交等真实管理科学问题时，仿真器的复杂度会急剧上升：需要同时处理数十辆车的运动轨迹、乘客的上下车流程、订单的状态流转、路径规划、需求生成，以及调度决策本身。

如果把所有逻辑都写在一个类或一个函数里，代码会迅速膨胀为难以维护的混乱结构。车辆移动的物理计算与订单匹配的业务规则交织在一起，调度算法与状态转移的边界模糊不清，任何一个小改动都可能引发连锁反应。更重要的是，当我们需要把调度策略从启发式切换为RL策略时，不得不在大量无关代码中寻找修改点。

本节介绍一种经过实践检验的仿真架构：将仿真器划分为算法层、逻辑层和物理层三个层次，每一层承担明确的职责，层与层之间通过清晰的接口通信。这种分层架构源自生产级车队调度仿真系统的工程实践，能够有效管理复杂度，并为RL策略的嵌入提供天然的接口。

三层架构概览

三层架构的核心思想是关注点分离（separation of concerns）：把仿真系统中性质不同的计算分配到不同的层次中，每一层只处理自己职责范围内的事务。

物理层（Physics Layer）

负责纯粹的动力学计算。在车队调度的场景中，物理层的唯一任务是根据车辆当前的位置、速度和路径，推算下一个时间步的位置。它不知道订单是什么，也不关心调度决策，只处理运动学。

逻辑层（Logic Layer）

负责业务规则和状态管理。逻辑层检测事件（车辆到达目的地、新订单产生），执行状态转移（订单从等待变为运送中，乘客上车、下车），管理队列和容量约束。它是物理层和算法层之间的桥梁：从物理层接收事件，向算法层提供状态查询，将算法层的决策转化为对实体的具体操作。

算法层（Algorithm Layer）

负责高层决策。在车队调度中，算法层的任务是给定当前可用车辆和待服务订单，决定哪辆车去服务哪个订单。RL策略或启发式算法就嵌入在这一层。算法层可以独立替换，而不需要修改物理层和逻辑层的代码。

物理层：纯粹的运动学

物理层是整个仿真器的底层引擎。它的职责极为单纯：根据车辆的当前位置、速度和预定路径，计算经过一个时间步 $\Delta t$ 后车辆的新位置。物理层不知道订单的存在，不关心调度策略，甚至不关心车辆为什么在移动。它只回答一个问题：按照当前速度沿当前路径走 $\Delta t$ 秒，车辆会到达哪里？

以下是物理层核心服务 VehiclePhysicsService 的实现。每辆车维护一条路径（由若干路径点组成的折线段），物理层沿折线段推进车辆位置。

class VehiclePhysicsService:
    def __init__(self, delta_t: float):
        self.delta_t = delta_t

    def update(self, vehicles: list[Vehicle]):
        """推进所有正在移动的车辆的位置"""
        for vehicle in vehicles:
            if vehicle.status != VehicleStatus.MOVING:
                continue
            if not vehicle.current_route:
                continue

            # 本时间步需要前进的距离：速度 × 时间步长
            distance_to_travel = vehicle.speed * self.delta_t
            self._advance_along_route(vehicle, distance_to_travel)

    def _advance_along_route(self, vehicle, distance):
        """沿折线路径前进指定距离"""
        route = vehicle.current_route
        while distance > 0 and vehicle.route_index < len(route) - 1:
            segment_start = route[vehicle.route_index]
            segment_end = route[vehicle.route_index + 1]
            segment_length = self._distance(segment_start, segment_end)
            remaining = segment_length - vehicle.segment_progress

            if distance >= remaining:
                # 走完当前线段，进入下一段
                distance -= remaining
                vehicle.route_index += 1
                vehicle.segment_progress = 0.0
            else:
                # 在当前线段内前进
                vehicle.segment_progress += distance
                distance = 0.0

        # 更新车辆坐标
        if vehicle.route_index >= len(route) - 1:
            vehicle.location = route[-1]
            vehicle.status = VehicleStatus.STOPPED
        else:
            vehicle.location = self._interpolate(
                route[vehicle.route_index],
                route[vehicle.route_index + 1],
                vehicle.segment_progress
            )

值得注意的是，当车辆走完整条路径时，物理层将其状态设为 STOPPED。但物理层并不处理到达后的业务逻辑（乘客上下车、订单状态更新等），这些属于逻辑层的职责。物理层只报告一个事实：车辆到达了路径的终点。

设计原则：物理层不包含任何 if order... 或 if schedule... 类型的判断。如果你发现物理层的代码里出现了业务概念，说明职责划分出了问题。物理层应当可以在完全不知道"订单""乘客""调度"等概念的情况下独立工作。

逻辑层：业务规则与状态管理

逻辑层是三层架构的枢纽。它从物理层接收事件（车辆停下了），根据业务规则执行相应操作（处理上下车、更新订单状态），并为算法层提供决策所需的状态信息（哪些车辆空闲、哪些订单在等待）。

逻辑层通常由多个服务类组成，每个服务类处理一类业务规则。以车队调度仿真器为例，核心的逻辑服务是 VehicleStateService，负责检测车辆到达目的地并执行相应的业务操作。

class VehicleStateService:
    """检测车辆到达目的地，执行取货和送货逻辑"""

    def update(self, vehicles, orders, current_time):
        for vehicle in vehicles:
            if vehicle.status != VehicleStatus.STOPPED:
                continue
            if not vehicle.schedule:
                vehicle.status = VehicleStatus.IDLE
                continue

            next_task = vehicle.schedule[0]
            if self._at_location(vehicle.location, next_task.location):
                self._process_arrival(vehicle, next_task,
                                      orders, current_time)

    def _process_arrival(self, vehicle, task, orders, current_time):
        # 处理取货：乘客上车，订单进入运送状态
        for order_id in task.pickup_order_ids:
            order = orders[order_id]
            vehicle.board_passenger(order.num_items)
            order.transition_to(OrderStatus.IN_TRANSIT, current_time)

        # 处理送货：乘客下车，订单完成
        for order_id in task.dropoff_order_ids:
            order = orders[order_id]
            vehicle.alight_passenger(order.num_items)
            order.transition_to(OrderStatus.COMPLETED, current_time)

        # 当前任务完成，从调度表中移除
        vehicle.schedule.pop(0)
        if vehicle.schedule:
            # 设置前往下一个任务地点的路径
            vehicle.current_route = self._plan_route(
                vehicle.location, vehicle.schedule[0].location
            )
            vehicle.route_index = 0
            vehicle.segment_progress = 0.0
            vehicle.status = VehicleStatus.MOVING
        else:
            vehicle.status = VehicleStatus.IDLE

逻辑层还可以包含其他服务，例如：

  • DemandService：按照预设的需求分布，在特定时刻生成新订单。
  • StopOperationService：处理车辆在站点停靠时的上下客时间模拟。
  • MetricsService：计算服务质量指标，如平均等待时间、完成率等。

这些服务各自独立，但共同维护着仿真系统的业务状态。它们之间通过共享的实体对象（Vehicle、Order）进行通信。

算法层：决策与策略

算法层位于架构的顶端，负责做出高层决策。在车队调度场景中，算法层的核心问题是：给定当前的空闲车辆集合和待服务订单集合，如何将车辆分配给订单？

算法层的入口是 DispatchController，它控制决策频率，并将具体的分配逻辑委托给可替换的策略对象。

class DispatchController:
    def __init__(self, strategy: DispatchStrategy):
        self.strategy = strategy   # RL策略或启发式
        self.decision_interval = 10.0  # 每10秒做一次决策
        self.last_decision_time = -float('inf')

    def update(self, vehicles, pending_orders, current_time):
        if current_time - self.last_decision_time < self.decision_interval:
            return []
        self.last_decision_time = current_time

        available = [v for v in vehicles
                     if v.status == VehicleStatus.IDLE]
        waiting = [o for o in pending_orders
                   if o.status == OrderStatus.WAITING]

        if not available or not waiting:
            return []

        return self.strategy.make_decisions(
            available, waiting, current_time
        )

策略的设计采用策略模式（Strategy Pattern）：定义统一的抽象接口，不同的策略实现这个接口。这样，切换算法只需要替换策略对象，而不需要修改控制器或仿真器的其他部分。

from abc import ABC, abstractmethod

class DispatchStrategy(ABC):
    @abstractmethod
    def make_decisions(self, vehicles, orders, current_time):
        """返回 [(vehicle_id, order_id), ...] 形式的分配列表"""
        pass


class NearestNeighborStrategy(DispatchStrategy):
    """最近邻启发式：将每个订单分配给距离最近的空闲车辆"""

    def make_decisions(self, vehicles, orders, current_time):
        decisions = []
        assigned_vehicles = set()
        for order in orders:
            best_vehicle = None
            best_dist = float('inf')
            for vehicle in vehicles:
                if vehicle.vehicle_id in assigned_vehicles:
                    continue
                dist = distance(vehicle.location, order.origin)
                if dist < best_dist:
                    best_dist = dist
                    best_vehicle = vehicle
            if best_vehicle:
                decisions.append(
                    (best_vehicle.vehicle_id, order.order_id)
                )
                assigned_vehicles.add(best_vehicle.vehicle_id)
        return decisions


class RLStrategy(DispatchStrategy):
    """RL策略：使用训练好的神经网络做调度决策"""

    def __init__(self, policy_network):
        self.policy = policy_network

    def make_decisions(self, vehicles, orders, current_time):
        obs = self._build_observation(vehicles, orders)
        actions = self.policy.predict(obs)
        return self._decode_actions(actions, vehicles, orders)

从启发式算法切换到RL策略，只需在初始化时更换策略对象：

# 使用启发式
controller = DispatchController(NearestNeighborStrategy())

# 切换为RL策略
controller = DispatchController(RLStrategy(trained_policy))

仿真器的其余代码（物理层、逻辑层）完全不需要修改。这正是分层架构的价值所在。

接口隔离：读与写的分离

三层架构还隐含着一个重要的设计原则：算法层只能读取状态，不能直接修改实体。所有对实体状态的修改都必须经过逻辑层。

为什么这很重要？考虑一个没有接口隔离的场景：算法层在做调度决策时，直接把车辆状态改成了 MOVING，但没有设置路径。这辆车在物理层的更新中就会出现异常。如果算法层直接修改了订单状态，而跳过了逻辑层的上下车处理，车辆的载客数和订单状态就会不一致。

接口隔离原则

算法层通过只读接口查询系统状态，通过命令发出决策。逻辑层是唯一能修改实体状态的组件。这种分离确保实体始终处于合法状态，避免因算法层的直接修改而引入不一致性。

在代码中，这种分离可以通过定义查询接口和命令接口来实现。

class VehicleQueryInterface:
    """算法层通过此接口查询车辆状态（只读）"""

    def get_idle_vehicles(self) -> list[Vehicle]:
        """返回所有空闲车辆"""
        ...

    def get_vehicle_location(self, vid: str) -> Location:
        """查询指定车辆的当前位置"""
        ...

    def get_vehicle_load(self, vid: str) -> int:
        """查询指定车辆的当前载客数"""
        ...


class VehicleCommandInterface:
    """逻辑层通过此接口修改车辆状态（写入）"""

    def assign_schedule(self, vid: str, schedule: list):
        """为车辆分配新的调度计划"""
        ...

    def update_location(self, vid: str, location: Location):
        """更新车辆位置（由物理层调用）"""
        ...

算法层持有 VehicleQueryInterface 的引用，只能读取信息。当算法层做出分配决策后，它返回一组 (vehicle_id, order_id) 元组，由逻辑层负责将这些决策转化为实体的状态变更。这条规则看似增加了代码量，实际上大幅降低了调试难度：当实体出现非法状态时，只需要在逻辑层排查原因，而不必在整个代码库中搜索。

为什么需要分层

对于小型仿真器，分层架构可能显得过度设计。但当仿真器达到生产级别的规模（数十个实体类型、上百种业务规则），不分层的代码几乎不可能维护。下表总结了分层架构带来的具体收益。

收益 说明
算法可替换 RL策略和启发式共享同一接口，切换策略只需替换一个对象，无需修改仿真器的其他部分。在训练阶段使用RL策略，在基线对比实验中使用启发式，代码完全相同。
独立测试 物理层可以脱离算法层单独做单元测试，验证车辆移动逻辑的正确性。逻辑层可以用预设的事件序列测试状态转移是否正确。每一层的测试都不依赖其他层。
团队协作 不同的团队成员可以同时开发不同的层。开发RL算法的研究者不需要了解物理引擎的细节，负责仿真环境的工程师也不需要理解策略梯度的数学。
定位问题 当仿真器产生异常行为时，可以逐层排查。车辆位置计算错误是物理层的问题，订单状态不一致是逻辑层的问题，分配效率低是算法层的问题。
跨场景复用 同一个物理层可以用于网约车调度、外卖配送、需求响应公交等不同场景。只需更换逻辑层和算法层的实现。

完整的一个时间步

为了让三层架构的协作方式更加具体，我们完整追踪一个时间步内各层的执行过程。假设当前时刻为 $t = 120$ 秒，时间步长 $\Delta t = 1$ 秒。

时间步 $t = 120 \to 121$ 秒的执行流程
  1. 物理层：VehiclePhysicsService.update() 推进所有移动中车辆的位置。车辆 A 沿路径前进 $v \cdot \Delta t = 10$ 米。
  2. 物理层：车辆 A 恰好到达路径终点，物理层将其状态设为 STOPPED。
  3. 逻辑层：VehicleStateService.update() 检测到车辆 A 的状态为 STOPPED，且调度表中的下一个任务地点与当前位置一致。执行送货逻辑：乘客下车，订单 #42 状态变更为 COMPLETED。车辆 A 的调度表为空，状态变为 IDLE。
  4. 逻辑层：DemandService.update() 根据需求分布，在当前时刻生成新订单 #57。
  5. 算法层：DispatchController.update() 发现距离上次决策已超过 10 秒。查询空闲车辆（车辆 A、车辆 B），查询等待中的订单（订单 #57）。调用策略，决定将订单 #57 分配给车辆 B。
  6. 逻辑层：接收算法层的分配结果，为车辆 B 创建调度表（先去取货点，再去送货点），规划路径，将车辆 B 状态设为 MOVING。

注意执行顺序：先物理层，再逻辑层，最后算法层，然后逻辑层将算法层的决策落地。这个顺序保证了逻辑层在处理事件时已经拿到最新的物理状态，算法层在做决策时已经拿到最新的业务状态。

用伪代码表示，主循环的结构如下：

class FleetSimulator:
    def __init__(self, physics, logic_services, controller):
        self.physics = physics
        self.logic_services = logic_services  # 逻辑层的各个服务
        self.controller = controller          # 算法层

    def step(self, current_time):
        # 1. 物理层：推进车辆位置
        self.physics.update(self.vehicles)

        # 2. 逻辑层：处理事件和状态转移
        for service in self.logic_services:
            service.update(self.vehicles, self.orders, current_time)

        # 3. 算法层：做出调度决策
        decisions = self.controller.update(
            self.vehicles, self.orders, current_time
        )

        # 4. 逻辑层：将决策转化为实体操作
        self._apply_decisions(decisions)

整个仿真器的主循环只有四步，每一步对应架构中的一个明确职责。新加入项目的开发者可以快速理解系统的整体结构，然后深入到具体的层去了解细节。

本节小结

简单的仿真器可以用一个类完成所有功能。但当仿真系统的复杂度增长到生产级别时，缺乏架构的代码会迅速失控。本节介绍的三层架构提供了一种经过实践验证的组织方式：

  • 物理层处理纯粹的运动学计算，不包含任何业务逻辑。
  • 逻辑层负责业务规则、事件检测和状态转移，是系统的枢纽。
  • 算法层负责高层决策，通过策略模式支持RL与启发式的无缝切换。

层间通过接口隔离保持边界清晰：算法层只读取状态、发出决策命令，逻辑层负责所有的状态修改。这种分离使得每一层可以独立开发、独立测试、独立迭代。

对于强化学习研究而言，三层架构的最大价值在于为RL策略提供了天然的嵌入点。算法层的策略接口就是RL智能体与环境的交互边界。训练阶段，RL策略在算法层中探索和学习；部署阶段，训练好的策略替换启发式，仿真器的其余部分无需任何修改。这正是第九章实际案例中采用的架构模式。

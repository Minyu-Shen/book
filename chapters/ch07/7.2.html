7.2 面向对象编程与智能体建模

为什么仿真器需要面向对象编程

考虑一个城市配送车队调度系统的仿真器。这个系统至少涉及以下实体：车辆（有位置、容量、载货量、行驶状态）、订单（有起点、终点、生成时间、当前状态）、站点（有坐标、等待队列）、路线（有途经站点序列、预计用时）。每种实体都有自己的数据和行为规则。例如，车辆需要检查是否超载才能装货；订单需要按照固定的生命周期流转状态；路线需要根据交通状况更新预计到达时间。

如果用纯过程式的写法，所有数据会散落在各种列表和字典中，所有行为逻辑混杂在一个庞大的函数里。当实体类型增多、交互关系变复杂时，代码很快变得难以维护。更危险的是，过程式代码中很难保证数据的一致性。例如，一段代码给车辆增加了货物，另一段代码却忘记更新载货量，导致仿真结果出错。

面向对象编程（Object-Oriented Programming, OOP）为这类问题提供了自然的解决方案。它的核心思想是：将数据和操作数据的行为打包在一起，形成对象。每个现实世界的实体对应一个类（class），类定义了该实体拥有的属性（数据）和方法（行为）。对象是类的实例，代表一个具体的实体（如编号为 $\text{V-001}$ 的车辆）。

OOP 的好处在于：

  • 模块化：每种实体的逻辑封装在自己的类中，互不干扰。修改车辆的逻辑不会影响订单的代码。
  • 一致性保护：通过方法来修改数据，可以在方法内部检查约束条件（如容量限制），防止实体进入非法状态。
  • 可复用：设计好的类可以在不同仿真场景中复用。例如，同一个 $\text{Vehicle}$ 类既可以用于配送仿真，也可以用于网约车仿真。
  • 可读性：代码的组织结构与问题的实体结构一一对应，便于理解和沟通。

本节将以城市配送车队调度为例，展示如何用 OOP 的方式建模仿真中的各类实体。

实体设计：车辆类

车辆是配送仿真中最核心的实体。一辆车在仿真过程中需要维护以下信息：

  • 标识：车辆编号 $\text{vehicle\_id}$
  • 位置：当前坐标 $(x, y)$
  • 容量与载货：最大容量 $\text{capacity}$，当前载货量 $\text{current\_load}$
  • 状态：空闲（$\text{IDLE}$）、行驶中（$\text{MOVING}$）、停靠中（$\text{STOPPED}$）
  • 任务列表：待执行的取货/送货任务序列 $\text{schedule}$

在 Python 中，我们使用 Enum 来定义有限的状态集合，使用 dataclass 来简洁地定义纯数据结构，并通过类的方法来保护状态变更的合法性。

from enum import Enum
from dataclasses import dataclass

class VehicleStatus(Enum):
    IDLE = "idle"
    MOVING = "moving"
    STOPPED = "stopped"

@dataclass
class Location:
    x: float
    y: float

class Vehicle:
    def __init__(self, vehicle_id: str, location: Location, capacity: int):
        self.vehicle_id = vehicle_id
        self.location = location
        self.capacity = capacity
        self.current_load = 0
        self.status = VehicleStatus.IDLE
        self.schedule: list = []

    def can_board(self, num_passengers: int) -> bool:
        """检查是否还有足够容量"""
        return self.current_load + num_passengers <= self.capacity

    def board_passenger(self, num_passengers: int):
        """装载货物/乘客，装载前检查容量约束"""
        if not self.can_board(num_passengers):
            raise ValueError(
                f"超出容量: {self.current_load}+{num_passengers} > {self.capacity}"
            )
        self.current_load += num_passengers

    def alight_passenger(self, num_passengers: int):
        """卸载货物/乘客，卸载前检查合法性"""
        if num_passengers > self.current_load:
            raise ValueError(f"下车人数超出载客量")
        self.current_load -= num_passengers

上述代码有几个值得注意的设计。

第一，VehicleStatus 使用 Enum 而非字符串。如果用字符串表示状态（如 "idle"、"moving"），拼写错误不会报错，调试时极难发现。使用 Enum 后，写错状态名会立即抛出 AttributeError。

第二，Location 使用 @dataclass。它本质上是一个只包含数据的容器，不需要复杂的行为逻辑。@dataclass 装饰器自动生成 __init__、__repr__ 等方法，比手写更简洁。

第三，board_passenger 和 alight_passenger 方法在修改 $\text{current\_load}$ 之前先做合法性检查。这是 OOP 最重要的设计原则之一，我们将在后文专门讨论。

实体设计：订单类与状态机

订单是仿真中另一个核心实体。与车辆不同，订单有一个明确的生命周期：从用户发出请求开始，经过派单、取货、配送，最终完成或被取消。这种具有固定状态流转规则的模式称为有限状态机（Finite State Machine, FSM）。

订单的状态机定义如下。订单在任意时刻处于以下五个状态之一：

  • $\text{WAITING}$：等待派单
  • $\text{ASSIGNED}$：已派单，等待取货
  • $\text{IN\_TRANSIT}$：运输中
  • $\text{COMPLETED}$：已送达
  • $\text{CANCELLED}$：已取消

合法的状态转移路径是固定的：$\text{WAITING}$ 可以转移到 $\text{ASSIGNED}$ 或 $\text{CANCELLED}$；$\text{ASSIGNED}$ 可以转移到 $\text{IN\_TRANSIT}$ 或 $\text{CANCELLED}$；$\text{IN\_TRANSIT}$ 只能转移到 $\text{COMPLETED}$；而 $\text{COMPLETED}$ 和 $\text{CANCELLED}$ 是终止状态，不能再转移。任何不在上述列表中的转移都应被视为程序错误。

图 7.2-1   订单生命周期状态机。实线为正常流转路径，虚线为取消路径。$\text{COMPLETED}$ 和 $\text{CANCELLED}$ 为终止状态。

在代码中，我们用一个字典 VALID_TRANSITIONS 显式列出所有合法的状态转移，并在 transition_to 方法中检查转移是否合法。这种做法将状态机的规则集中定义在一处，避免了在代码各处散布 if-else 判断。

class OrderStatus(Enum):
    WAITING = "waiting"
    ASSIGNED = "assigned"
    IN_TRANSIT = "in_transit"
    COMPLETED = "completed"
    CANCELLED = "cancelled"

class Order:
    # 显式定义所有合法的状态转移
    VALID_TRANSITIONS = {
        OrderStatus.WAITING:    {OrderStatus.ASSIGNED, OrderStatus.CANCELLED},
        OrderStatus.ASSIGNED:   {OrderStatus.IN_TRANSIT, OrderStatus.CANCELLED},
        OrderStatus.IN_TRANSIT: {OrderStatus.COMPLETED},
        OrderStatus.COMPLETED:  set(),
        OrderStatus.CANCELLED:  set(),
    }

    def __init__(self, order_id, origin, destination, request_time, num_items=1):
        self.order_id = order_id
        self.origin = origin           # Location 对象
        self.destination = destination  # Location 对象
        self.request_time = request_time
        self.num_items = num_items
        self.status = OrderStatus.WAITING
        # 用于计算 KPI 的时间戳
        self.assignment_time = None
        self.pickup_time = None
        self.delivery_time = None

    def transition_to(self, new_status: OrderStatus, current_time: float):
        """将订单转移到新状态，同时记录时间戳"""
        if new_status not in self.VALID_TRANSITIONS[self.status]:
            raise ValueError(
                f"非法状态转移: {self.status.value} → {new_status.value}"
            )
        self.status = new_status
        # 自动记录各阶段时间戳
        if new_status == OrderStatus.ASSIGNED:
            self.assignment_time = current_time
        elif new_status == OrderStatus.IN_TRANSIT:
            self.pickup_time = current_time
        elif new_status == OrderStatus.COMPLETED:
            self.delivery_time = current_time

    @property
    def wait_time(self):
        """计算从请求到被分配的等待时间"""
        if self.assignment_time is not None:
            return self.assignment_time - self.request_time
        return None

    @property
    def service_time(self):
        """计算从取货到送达的服务时间"""
        if self.pickup_time is not None and self.delivery_time is not None:
            return self.delivery_time - self.pickup_time
        return None

这段代码的关键设计在于 transition_to 方法承担了两项职责：验证状态转移的合法性，以及自动记录对应的时间戳。外部代码只需调用 order.transition_to(OrderStatus.ASSIGNED, t)，不需要分别设置状态和时间戳。这确保了状态和时间戳始终保持一致。

此外，$\text{wait\_time}$ 和 $\text{service\_time}$ 用 @property 装饰器定义为只读属性。它们的值由其他属性派生而来，而非独立存储，避免了数据冗余带来的不一致风险。以 $\text{wait\_time}$ 为例，它的计算规则为：

$$ \text{wait\_time} = \text{assignment\_time} - \text{request\_time} \tag{7.2-1} $$

如果将 $\text{wait\_time}$ 作为独立属性存储，每当 $\text{assignment\_time}$ 更新时都需要同步更新 $\text{wait\_time}$，一旦遗漏就会产生不一致。而用 @property 实现的派生属性总是实时计算，从根源上避免了这个问题。

为什么显式定义状态转移？ 在仿真调试中，最难排查的 bug 往往是实体进入了非法状态。例如，一个已经送达的订单被再次派单，或一个尚未取货的订单被标记为送达。如果没有状态转移检查，这类错误可能不会立即报错，而是在很久之后才以不合理的 KPI 数值暴露出来，届时已很难追溯根因。显式定义 VALID_TRANSITIONS 可以让这类错误在发生的瞬间就被捕获。

设计原则：封装与受保护的状态更新

回顾上面的 Vehicle 类。假设调度逻辑中需要给车辆装载 $3$ 件货物，有两种写法：

直接修改属性

vehicle.current_load += 3

跳过容量检查，可能导致 $\text{current\_load} > \text{capacity}$

通过方法修改

vehicle.board_passenger(3)

方法内部自动检查容量，违规时立即报错

左边的写法绕过了容量约束检查。如果仿真中某处代码不小心给一辆容量为 $5$ 的车装了 $8$ 件货物，程序不会报错，但仿真结果将是错误的。这种 bug 的隐蔽性极强，可能在调试数天后才被发现。

右边的写法将修改操作封装在 board_passenger 方法中。方法在修改前先调用 can_board 检查容量约束。如果违反约束，立即抛出异常，开发者可以迅速定位问题。

这个道理类似于银行 ATM 的设计：ATM 不会让你直接修改账户余额，而是通过取款操作来间接修改，取款操作会先验证余额是否充足。这种设计保证了系统的不变量（invariant），即在任何时刻都必须成立的约束条件。对于车辆而言，不变量是：

$$ 0 \leq \text{current\_load} \leq \text{capacity} \tag{7.2-2} $$

对于订单而言，不变量包括：状态只能按 $\text{VALID\_TRANSITIONS}$ 中定义的路径流转；$\text{assignment\_time}$ 非空当且仅当状态已经经过 $\text{ASSIGNED}$。

封装的核心目的

封装不仅仅是隐藏数据，更重要的是保护不变量。每一次状态变更都通过方法进行，方法在修改前检查约束条件，确保实体永远不会进入非法状态。在仿真开发中，这意味着 bug 能够在发生的第一时间被发现，而不是在数千步仿真之后才以不合理的结果暴露出来。

集合管理：从单个实体到车队

仿真中通常需要管理大量同类实体。例如，一个配送系统可能有数十辆车、数百个订单同时存在。我们需要一种方式来组织这些实体，并提供高效的查询能力。

最直观的做法是使用字典，以实体的唯一标识（如车辆编号）为键：

class Fleet:
    def __init__(self):
        self.vehicles: dict[str, Vehicle] = {}

    def add_vehicle(self, vehicle: Vehicle):
        """添加车辆到车队"""
        self.vehicles[vehicle.vehicle_id] = vehicle

    def get_idle_vehicles(self) -> list[Vehicle]:
        """返回所有空闲车辆"""
        return [v for v in self.vehicles.values()
                if v.status == VehicleStatus.IDLE]

    def get_available_vehicles(self, num_passengers: int) -> list[Vehicle]:
        """返回所有有足够容量的车辆"""
        return [v for v in self.vehicles.values()
                if v.can_board(num_passengers)]

Fleet 类将车辆集合的管理逻辑封装在一处。外部代码不需要知道车辆内部如何存储，只需通过 get_idle_vehicles() 等方法获取所需的车辆子集。这些查询方法直接复用了 Vehicle 类上已有的方法（如 can_board），体现了 OOP 的层次化组织优势。

类似地，我们可以设计一个 OrderPool 来管理所有订单：

class OrderPool:
    def __init__(self):
        self.orders: dict[str, Order] = {}

    def add_order(self, order: Order):
        """添加订单到订单池"""
        self.orders[order.order_id] = order

    def get_pending_orders(self) -> list[Order]:
        """返回所有等待派单的订单"""
        return [o for o in self.orders.values()
                if o.status == OrderStatus.WAITING]

    def get_completed_orders(self) -> list[Order]:
        """返回所有已完成的订单"""
        return [o for o in self.orders.values()
                if o.status == OrderStatus.COMPLETED]

    def average_wait_time(self) -> float:
        """计算已完成订单的平均等待时间"""
        completed = self.get_completed_orders()
        if not completed:
            return 0.0
        wait_times = [o.wait_time for o in completed
                      if o.wait_time is not None]
        return sum(wait_times) / len(wait_times) if wait_times else 0.0

注意 average_wait_time 方法如何利用 Order 类的 $\text{wait\_time}$ 属性来计算 KPI。这是一个两层调用的例子：OrderPool 负责筛选和聚合，Order 负责计算单个订单的等待时间。每一层只做自己该做的事情，职责划分清晰。

实体间的协作

在实际的仿真逻辑中，不同实体需要协作。例如，当调度系统将一个订单分配给一辆车时，需要同时更新订单的状态和车辆的载货量。下面展示一个简化的调度函数：

def assign_order_to_vehicle(order: Order, vehicle: Vehicle,
                            current_time: float):
    """将订单分配给车辆"""
    # 1. 检查车辆容量
    if not vehicle.can_board(order.num_items):
        raise ValueError(
            f"车辆 {vehicle.vehicle_id} 容量不足，"
            f"无法装载订单 {order.order_id}"
        )
    # 2. 更新订单状态（内部检查状态转移合法性）
    order.transition_to(OrderStatus.ASSIGNED, current_time)
    # 3. 装载货物（内部再次检查容量）
    vehicle.board_passenger(order.num_items)
    # 4. 将任务添加到车辆调度表
    vehicle.schedule.append(order.order_id)

这个函数的逻辑非常清晰，因为每个实体已经在自己的方法中封装了验证逻辑。调度函数不需要知道容量检查的具体规则，也不需要知道哪些状态转移是合法的，它只需要调用已有的方法即可。如果任何一步检查失败（如车辆容量不足，或订单当前状态不允许被分配），相应的方法会抛出异常，阻止后续步骤执行。

注意操作的原子性。 在上面的函数中，如果第 3 步（装载货物）失败，第 2 步（订单状态变更）已经执行了。这意味着订单状态变成了 $\text{ASSIGNED}$，但货物没有装上车。在生产级仿真器中，需要考虑事务性，即要么所有步骤都成功，要么全部回滚。一种简单的做法是先做所有检查，再做所有修改：

# 先检查所有前置条件
assert vehicle.can_board(order.num_items)
assert OrderStatus.ASSIGNED in Order.VALID_TRANSITIONS[order.status]

# 再执行所有修改
order.transition_to(OrderStatus.ASSIGNED, current_time)
vehicle.board_passenger(order.num_items)
vehicle.schedule.append(order.order_id)

完整示例：创建并操作仿真实体

下面的代码展示了如何将以上所有类组合起来，完成一个简单的调度流程：

# 创建车辆
v1 = Vehicle("V-001", Location(0.0, 0.0), capacity=5)
v2 = Vehicle("V-002", Location(3.0, 4.0), capacity=3)

# 创建车队并添加车辆
fleet = Fleet()
fleet.add_vehicle(v1)
fleet.add_vehicle(v2)

# 创建订单
o1 = Order("O-101", origin=Location(1.0, 1.0),
           destination=Location(5.0, 5.0),
           request_time=0.0, num_items=2)
o2 = Order("O-102", origin=Location(2.0, 3.0),
           destination=Location(6.0, 1.0),
           request_time=1.5, num_items=4)

# 在 t=2.0 时刻进行调度
current_time = 2.0

# 将 O-101 分配给 V-001
assign_order_to_vehicle(o1, v1, current_time)
print(f"订单 {o1.order_id}: {o1.status.value}")
# 输出: 订单 O-101: assigned
print(f"车辆 {v1.vehicle_id}: 载货 {v1.current_load}/{v1.capacity}")
# 输出: 车辆 V-001: 载货 2/5

# 尝试将 O-102 分配给 V-002（容量为 3，但需要装载 4 件）
try:
    assign_order_to_vehicle(o2, v2, current_time)
except ValueError as e:
    print(f"分配失败: {e}")
# 输出: 分配失败: 车辆 V-002 容量不足，无法装载订单 O-102

# 在 t=5.0 时刻，O-101 取货完成，开始运输
o1.transition_to(OrderStatus.IN_TRANSIT, 5.0)

# 在 t=8.0 时刻，O-101 送达
o1.transition_to(OrderStatus.COMPLETED, 8.0)
v1.alight_passenger(o1.num_items)

print(f"订单 {o1.order_id} 等待时间: {o1.wait_time}")
# 输出: 订单 O-101 等待时间: 2.0
print(f"订单 {o1.order_id} 服务时间: {o1.service_time}")
# 输出: 订单 O-101 服务时间: 3.0

这段代码展示了实体类在实际使用中的样貌。每一步操作都有清晰的语义，且错误会被立即捕获。当仿真规模增大到数百辆车、数千个订单时，这种结构化的代码组织方式将显著降低调试和维护的成本。

从实体类到仿真器

本节介绍的 Vehicle、Order、Fleet、OrderPool 等类是仿真器的基础构件。它们定义了仿真中各类实体的数据结构和行为规则，但还没有涉及仿真的核心问题：时间如何推进？每个时间步里发生什么？实体之间如何按照时序进行交互？

这正是 5.3 节将要解决的问题。在 5.3 节中，我们将引入基于时间步的仿真循环，把这些实体类放入一个随时间推进的仿真框架中。在 5.4 节中，我们将进一步把仿真器组织为算法层、逻辑层、物理层的分层架构，使 RL 智能体可以方便地与仿真器交互。

好的实体设计让仿真逻辑变得简单。如果 Vehicle.board_passenger() 已经处理了所有容量验证，仿真循环就不需要操心容量检查；如果 Order.transition_to() 已经强制了状态转移规则，调度逻辑就不需要记住哪些转移是合法的。实体类承担了约束守卫的角色，让上层代码可以专注于业务逻辑本身。

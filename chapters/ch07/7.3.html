7.3 基于时间步的仿真逻辑

5.2 节介绍了如何用面向对象的方式定义仿真中的实体。有了实体类之后，下一个问题是：如何让这些实体在时间维度上运动起来？这正是仿真循环的职责。仿真循环是任何仿真器的核心骨架，它驱动仿真时钟一步步前进，在每一步更新所有实体的状态，并在适当的时刻请求决策算法给出动作。

在基于时间步的仿真中，仿真器维护一个全局时钟 $t$，从初始时刻 $t = 0$ 出发，以固定步长 $\Delta t$ 向前推进：

$$t = 0, \; \Delta t, \; 2\Delta t, \; 3\Delta t, \; \ldots$$

每推进一步，仿真器依次完成若干子任务：更新物理状态、处理逻辑事件、请求决策、执行动作、计算奖励。当时钟到达终止时刻 $T_{\text{end}}$ 时，一个回合结束。从 MDP 的视角看，仿真循环正是转移函数 $P(s' \mid s, a)$ 的计算实现：给定当前状态 $s$ 和动作 $a$，通过一次循环迭代，仿真器生成下一个状态 $s'$ 和对应的奖励 $r$。

仿真循环的基本结构

一个完整的仿真循环在每个时间步内执行以下四个阶段：

仿真循环伪代码
  1. 物理更新：根据运动学方程移动车辆，更新位置和速度。
  2. 逻辑更新：检测到达事件、生成新需求、更新实体状态标记。
  3. 决策请求：将当前环境状态传给决策算法，获取动作。
  4. 执行与奖励：执行调度决策，计算该时间步的即时奖励。

时钟推进：$t \leftarrow t + \Delta t$，若 $t \geq T_{\text{end}}$ 则终止。

简单示例：库存管理仿真器

在展示复杂的多服务仿真引擎之前，先用一个简单的例子来理解仿真循环的基本结构。以 5.1 节的库存管理问题为例，下面的代码直接实现了仿真循环，而不依赖任何第三方框架。

库存管理仿真器维护两个状态变量：当前库存量 $I_t$ 和时间步 $t$。在每个时间步，仿真器依次执行补货、需求采样、奖励计算和库存更新。即时奖励的计算公式为：

$$R_t = 10 \cdot \min(I_t, D_t) - 1 \cdot \max(I_t - D_t, 0) - 5 \cdot \max(D_t - I_t, 0) - 2 \cdot a_t \tag{7.3-2}$$

其中 $D_t \sim \text{Poisson}(\lambda = 5)$ 是随机需求，$a_t$ 是补货量，四项分别对应销售收入、持有成本、缺货惩罚和补货成本。

import numpy as np

class InventorySimulator:
    def __init__(self, horizon=30, max_inventory=20, seed=None):
        self.horizon = horizon
        self.max_inventory = max_inventory
        self.rng = np.random.default_rng(seed)

    def reset(self):
        """重置仿真：初始库存为 10，时间步归零。"""
        self.inventory = 10
        self.time = 0
        return self._get_state()

    def step(self, action):
        """执行一个时间步。action 为补货量。"""
        # 1. 补货（库存不超过上限）
        self.inventory = min(self.inventory + action, self.max_inventory)

        # 2. 随机需求
        demand = self.rng.poisson(lam=5)

        # 3. 计算奖励
        sold = min(self.inventory, demand)
        holding = max(self.inventory - demand, 0)
        shortage = max(demand - self.inventory, 0)
        reward = 10.0 * sold - 1.0 * holding - 5.0 * shortage - 2.0 * action

        # 4. 状态转移
        self.inventory = max(self.inventory - demand, 0)
        self.time += 1
        done = (self.time >= self.horizon)
        return self._get_state(), reward, done

    def _get_state(self):
        return {"inventory": self.inventory, "time": self.time}

这段代码虽然简短，但已经体现了仿真循环的完整结构。step 方法中的四个步骤（补货、需求、奖励、转移）对应图 7.3-1 中的四个阶段。使用时只需在外层编写循环：

sim = InventorySimulator(seed=42)
state = sim.reset()
total_reward = 0.0

while True:
    action = 5  # 简单策略：每步补货 5 件
    state, reward, done = sim.step(action)
    total_reward += reward
    if done:
        break

print(f"累计奖励: {total_reward:.1f}")

复杂示例：车队调度仿真引擎

库存管理的仿真循环足够简单，一个 step 方法就能容纳所有逻辑。但在车队调度这类复杂场景中，物理更新（车辆移动）、逻辑更新（到达检测、需求生成）、决策控制（派单算法）涉及大量代码，如果全部塞进一个方法会导致维护困难。更好的做法是将各阶段拆分为独立的服务（Service），由仿真引擎按固定顺序逐一调用。

class SimulationEngine:
    def __init__(self, config):
        self.time = 0.0
        self.end_time = config["end_time"]
        self.delta_t = config["delta_t"]  # 例如 1.0 秒

        # 初始化各服务（5.4 节会详细介绍分层架构）
        self.physics_service = VehiclePhysicsService(self.delta_t)
        self.demand_service = DemandService(config["demand_data"])
        self.vehicle_state_service = VehicleStateService()
        self.dispatch_controller = DispatchController()

        self.vehicles = []  # 由外部注入车辆实体列表

    def run(self):
        """主循环：驱动仿真从起始时刻运行到终止时刻。"""
        while self.time < self.end_time:
            self._tick()
            self.time += self.delta_t

    def _tick(self):
        """单个时间步：按固定顺序调用各服务。"""
        # 1. 物理更新：移动车辆
        self.physics_service.update(self.vehicles, self.time)

        # 2. 逻辑更新：检测到达、生成新需求
        self.demand_service.update(self.time)
        self.vehicle_state_service.update(self.vehicles, self.time)

        # 3. 控制决策：调度算法做决策
        decisions = self.dispatch_controller.update(
            self.vehicles, self.demand_service.pending_orders, self.time
        )

        # 4. 执行决策
        self._apply_decisions(decisions)

与库存管理的单方法实现相比，仿真引擎的 _tick 方法更像一个调度器：它本身不包含具体的业务逻辑，只负责按正确的顺序调用各服务的 update 方法。各服务内部独立管理自己的状态，引擎只关心调用顺序。这种设计使得修改某个服务（例如更换需求模型）不会影响其他服务的代码。5.4 节将进一步讨论这种分层架构的设计原则。

执行顺序的重要性

仿真循环中各阶段的执行顺序不是随意的，颠倒顺序可能引入难以发现的 bug。下面通过一个具体场景说明这一点。

假设某辆车在 $t = 100\text{s}$ 时距离目的地还有 $5\text{m}$，而 $\Delta t = 1\text{s}$，车速为 $10\text{m/s}$。在正确的执行顺序下：

  1. 物理更新先执行：车辆前进 $10\text{m}$，抵达目的地。
  2. 逻辑更新检测到到达事件，将车辆标记为空闲。
  3. 决策阶段看到一辆空闲车辆，可以将它分配给新订单。

如果颠倒前两步的顺序：

  1. 逻辑更新先执行：车辆还在距目的地 $5\text{m}$ 处，未到达，不标记空闲。
  2. 物理更新执行：车辆前进 $10\text{m}$，抵达目的地。
  3. 决策阶段看不到空闲车辆（因为逻辑更新已经结束），导致新订单无法被及时分配。

这辆车要等到下一个时间步才会被标记为空闲，白白浪费了一个 $\Delta t$ 的调度窗口。在高峰期，这种延迟会显著降低服务质量。

常见陷阱   仿真循环内各阶段的执行顺序必须固定且合理。经验法则是：先让物理世界前进（物理更新），再检测物理变化带来的逻辑后果（逻辑更新），最后基于最新的世界状态做决策（决策请求）。颠倒顺序不会导致程序报错，但会产生时序偏差，使仿真结果系统性地偏离预期。

时间步长 $\Delta t$ 的选择

时间步长 $\Delta t$ 是仿真器最重要的超参数之一。它直接影响仿真的精度和计算速度。整个仿真的总步数为：

$$N_{\text{steps}} = \frac{T_{\text{end}} - T_{\text{start}}}{\Delta t} \tag{7.3-1}$$

以一个持续 4 小时（14400 秒）的仿真为例，不同的 $\Delta t$ 对应的步数差异如下表所示。

$\Delta t$总步数适用场景
$1\text{s}$14,400需要精确模拟车辆运动轨迹
$10\text{s}$1,440兼顾精度与速度的常用选择
$60\text{s}$240宏观调度策略的快速验证

$\Delta t$ 的选择需要在两个方面做权衡。

精度方面，$\Delta t$ 应当小于仿真中最快动态过程的特征时间。例如，如果路网中最短的路段行驶时间是 10 秒，那么 $\Delta t$ 应当远小于 10 秒，否则车辆可能在一个时间步内直接跳过中间的路段节点，产生不合理的位置跳变。

速度方面，每个时间步都需要遍历所有实体并更新状态。$\Delta t$ 减半意味着步数翻倍，仿真所需的计算时间也大致翻倍。在 RL 训练中，仿真器需要反复运行数千乃至数万个回合，仿真速度直接影响训练效率。

实践建议   可以先用较小的 $\Delta t$ 建立一个高精度的参考仿真，然后逐步增大 $\Delta t$，观察关键指标（如平均等待时间、车辆利用率）的变化。当指标的偏差在可接受范围内时，就找到了合适的 $\Delta t$。

仿真时间步与决策时间步

在复杂仿真中，RL 智能体不需要在每个仿真时间步都做决策。仿真引擎的 step 方法可以在内部推进多个仿真时间步，直到遇到一个需要决策的时刻才返回。

class FleetSimulator:
    def step(self, action):
        # 将 RL 动作传入调度控制器
        self.dispatch_controller.set_action(action)

        # 推进仿真到下一个决策时刻
        reward = 0.0
        while not self._needs_decision():
            self._tick()
            self.time += self.delta_t
            reward += self._get_step_reward()
            if self._is_done():
                break

        state = self._get_state()
        done = self._is_done()
        return state, reward, done

    def _needs_decision(self):
        """判断当前是否需要 RL 智能体介入决策。"""
        return (self.dispatch_controller.has_pending_orders()
                or self._is_done())

这种设计的关键在于：RL 智能体的一次动作可能跨越多个仿真时间步，奖励在此期间累加。这符合第 4.3 节讨论的事件触发型决策：智能体不必在每个时间步都介入，只需要在有意义的时刻（例如有新订单到达或有车辆变为空闲时）做决策。

仿真时间步与决策时间步的区别

仿真时间步由 $\Delta t$ 决定，是仿真器内部的物理时钟粒度。决策时间步由业务逻辑决定，对应 RL 算法的一次 step 调用。两者的比值可以是动态的：在事件密集的时段，决策间隔可能只有 $1 \cdot \Delta t$；在空闲时段，可能相隔 $100 \cdot \Delta t$。

确定性重放与随机种子

仿真器中的随机性来源于需求生成、行驶时间扰动等多个环节。在调试和实验对比中，确定性重放是一项不可或缺的能力：给定相同的随机种子和相同的动作序列，仿真器必须产生完全一致的轨迹。

实现确定性重放的关键是统一管理随机数生成器。常见做法是在仿真引擎的 reset 方法中接受一个种子参数，并用它初始化一个全局的随机数生成器 rng。所有需要随机数的服务都从这个 rng 派生子生成器，而不是各自调用 random 或 np.random 的全局状态。

class SimulationEngine:
    def reset(self, seed=None):
        """重置仿真状态，初始化随机数生成器。"""
        self.rng = np.random.default_rng(seed)
        self.time = 0.0

        # 为各服务分配独立的子生成器
        self.demand_service.rng = self.rng.spawn(1)[0]
        self.physics_service.rng = self.rng.spawn(1)[0]

        # 重置各服务状态
        self.demand_service.reset()
        self.physics_service.reset()
        self.vehicle_state_service.reset()

NumPy 的 default_rng 提供了 spawn 方法，可以从一个父生成器派生出多个相互独立的子生成器。这样即使各服务消耗随机数的顺序发生变化（例如新增了一个服务），也不会影响其他服务收到的随机数序列。

调试技巧   当 RL 训练出现异常时，可以记录出问题的回合所使用的种子和动作序列，然后用确定性重放逐步检查仿真状态。这比在随机环境中反复重跑要高效得多。

本节小结

本节介绍了基于时间步的仿真循环的核心逻辑，要点如下：

  • 仿真循环是 MDP 转移函数 $P(s' \mid s, a)$ 的计算实现。每个时间步依次执行物理更新、逻辑更新、决策请求和动作执行四个阶段。
  • 各阶段的执行顺序必须固定且合理（先物理、再逻辑、后决策），颠倒顺序会引入时序偏差。
  • 时间步长 $\Delta t$ 需要在仿真精度和计算速度之间做权衡，应小于系统中最快动态过程的特征时间。
  • 仿真器的 step 方法可以跨越多个仿真时间步，在下一个决策时刻才返回，从而支持事件触发型决策。
  • 通过统一管理随机种子，仿真器可以实现确定性重放，这对调试和实验复现至关重要。

到目前为止，我们已经知道如何定义仿真实体（5.2 节）以及如何驱动它们在时间维度上演化（本节）。下一节将进一步讨论如何将仿真器组织为清晰的分层架构，使得算法层、逻辑层和物理层各司其职、互不耦合。
